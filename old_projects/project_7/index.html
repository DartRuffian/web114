  2. Use Comments.

    You may know what your code does when you're writing it; however, if you're working on a team of developers, some members may not to be able to understand it. Even if you are working by yourself, you may have to go back to older code to modify or fix code, and comments make understanding code much easier. Now, you don't need to have a comment for every single line of code, but for blocks of code, leave a comment or two describing what it does and how it does it.

  3. Use Meaningful Names

    For things like variables, functions, classes, etc. you'll want to use names that convey what they are used for. Single character names (in most cases) don't do this, names like x, y, don't always convey their meaning. These names can be used when it makes sense, such as keeping track of the posistion of something, for example, but in most cases, use a more meaningful name.

Identify three distinct sloppy programming practices and explain the problems they each create.

  1. "I'll fix it later"

    Delaying fixing a problem not only doesn't solve that specific problem, but is also typically represenative of a much larger problem. Where multiple fixes are pushed back to be fixed "later" and then forgotten.

  2. Always Using One-Liner Solution

    "One-Liners", blocks of complex code that can be reduce to one line of code, are clever; however, most of the time, they are unorganized and illegible to the everyone but the author. Take this Python One-Liner that uses the "Sieve of Eratosthenes" algorithm to find all prime numbers up to a given value: lambda n: functools.reduce( (lambda r,x: r-set(range(x**2,n,x)) if (x in r) else r), range(2,int(n**0.5)), set(range(2,n))). To someone who is not familiar with this specific algorithm, or even Python itself, this looks like a garbled mess. Remember that code is executed by machines but read by humans. Focus on making your code readable, and then optimize.

  3. You don't need to re-invent the wheel.

    You don't always have to write your own implentation of a type of system. You don't need to re-invent the wheel when someone else has the blueprints to a car. This however, does not mean that you should just copy and paste code without thinking. Consider on if that system will work for you, or perhaps make changes to it to better fit your needs.

Explain the difference between a primitive, an array, and an object and include an example of each.

A "primitive" data type are the data types that JavaScript predefines for us to use. Types like byte, int, boolean, etc. are all primitive. Object data types however, are user-created. Such as classes or strings are object data types and therefore created by the user. Arrays are a special type of data that can hold multiple values, all neatly rolled into one. After all, why create three variables for a set of values, when you can have a single array that contains all three values.

Explain the phrase "Fractions divided by powers of 2 are exactly represented in the binary system" and what universal problem this is referring to in all programming languages.

In many, many languages, if you were to add 0.1 and 0.2, you will not get 0.3 as you might expect, but instead get a much longer 0.30000000000000004, but why is this? But first let's explain this in the way that we humans express numbers. We humans use "Base 10" for math, meaning there are 10 numbers from 0 to 9 that make up all of the numbers we use; and with Base 10, you can express all of the fractions finitely that are Prime Factors of 10. So fractions like 1⁄2, 1⁄4, 1⁄5, 1⁄8, and 1⁄10 are all very easily expressed using Base 10.

What do we do about the fractions in-between though? How might we express 1⁄3 as a decimal? We could approximate and use 0.3, or we could get even closer and use 0.33, or even closer than that with 0.333, and so it goes on and on. Meaning that 1⁄3 cannot be expressed in Base 10.

Computers do this in much the same way that humans do; however, they use a Base 2 system, also known as binary, to express numbers. So similar to how 1⁄3 is a repeating decimal in Base 10, so is 1⁄10,1⁄5, etc. etc.
This is what leads to 0.1 and 0.2 adding to 0.30000000000000004.
